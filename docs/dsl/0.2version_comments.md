## Sophie DSL v0.2

### Overview

Sophie is a small DSL to express:

* **Trade rules** (BUY/SELL) optionally guarded by boolean conditions.
* **Portfolio allocations** as a sum of labeled positions.
* **Market expressions** over prices and historical series (fields + TA indicators).

The grammar is designed to be easy to parse (ANTLR) and easy to interpret in a later execution engine.

---

## Differences from v0.1

Version **0.2** is a compatibility + expressiveness update over **0.1**. The main user-visible changes are:

* **Symbols vs. currencies**: v0.1 required `OF IDENTIFIER` and `IDENTIFIER.DOT field`, but currency tickers like `BTC` are lexed as `CURRENCY`, not `IDENTIFIER`. v0.2 introduces `symbol : IDENTIFIER | CURRENCY` and uses it everywhere, so `OF BTC`, `BTC.volume`, and `PRICE(BTC)` are valid.

* **Optional conditions**: v0.1 required `IF condition` in every trade. v0.2 makes the clause optional with `(IF condition)?`, allowing unconditional trades.

* **Portfolio syntax now matches the examples**: v0.1 defined allocations as a sum of `value` only (e.g., `500 USD + 0.1 BTC`), but the examples used `value OF asset`. v0.2 introduces `allocation : value OF symbol` and makes `allocation_list` a sum of allocations.

* **Arithmetic is now part of the parser**: v0.1 had arithmetic tokens in the lexer but comparisons were between `operand`s only, so expressions like `PRICE(BTC) * 0.15` were not parseable. v0.2 adds `expr/term/primary` so arithmetic works inside comparisons.

* **Order size model extended**: v0.2 introduces `consideration` with `value` (money-based) or `QTY n` (explicit quantity). This was not available in v0.1.

* **Indicator window argument fixed**: v0.1 used `INTEGER` in `RSI(symbol, 14)`-style calls, but because `NUMBER` also matches integers and is typically chosen by the lexer, this could create a lexer/parser mismatch. v0.2 uses `NUMBER` for the window argument, accepting both `14` and `14.0`.

## Lexical design

### Tokens that matter

* **Keywords are reserved**: `BUY`, `SELL`, `OF`, `IF`, `PORTFOLIO`, `QTY`
* **Currencies are reserved**: `CURRENCY ∈ {EUR, USD, GBP, BTC}`
* **Identifiers**: `IDENTIFIER` for generic tickers/symbols (e.g. `AAPL`, `MSFT`, `VWCE`)
* **Numbers**: `NUMBER` matches integers and decimals (e.g. `42`, `3.14`)

### Key choice: `CURRENCY` vs `IDENTIFIER`

Currencies are their own token so they don’t collide with identifiers.
Because of that, `BTC` is lexed as `CURRENCY`, not as `IDENTIFIER`.

To still allow currency tickers to behave like normal tradable instruments, the parser introduces:

```antlr
symbol : IDENTIFIER | CURRENCY ;
```

So `BTC` can appear where a “ticker” is expected (e.g., `OF BTC`, `BTC.volume`, `PRICE(BTC)`).

## Program structure

A program is one or more statements, optionally separated by semicolons:

```ebnf
program := (statement ';'?) + EOF
statement := trade_cmd | portfolio_cmd
```

---

## Trade commands

### Syntax

```ebnf
trade_cmd := (BUY | SELL) consideration OF symbol (IF condition)?
consideration := value | quantity
value := NUMBER CURRENCY
quantity := 'QTY' NUMBER
```

### Meaning

* `value` represents a **monetary amount** (e.g. `1500 EUR`) that your runtime can later convert into quantity using prices.
* `QTY n` represents an **explicit quantity** (e.g. `QTY 5`) bypassing conversion.

### Optional condition

`IF condition` is optional: you can write unconditional trades.

Examples:

```text
BUY 1500 EUR OF MSFT
BUY QTY 2 OF AAPL IF PRICE(AAPL) > 150
SELL 0.5 BTC OF BTC IF BTC.volume > 1000000
```

---

## Portfolio assignment

### Syntax

```ebnf
portfolio_cmd := PORTFOLIO '=' allocation_list
allocation_list := allocation ('+' allocation)*
allocation := value OF symbol
```

### Meaning

A portfolio is a sum of target allocations, each expressed as **amount + currency + instrument**.

Example:

```text
PORTFOLIO =
  6000 EUR OF VWCE
  + 2000 USD OF AAPL
  + 0.1 BTC OF BTC
```

## Market expressions

Sophie supports arithmetic expressions for use inside comparisons.

### Arithmetic (numeric) expressions

```ebnf
comparison := expr (('>' | '<' | '=' | '!=') expr)?
expr := term (('+' | '-') term)*
term := primary (('*' | '/') primary)*
primary := NUMBER | price_expr | series_operation | '(' expr ')'
```

This enables formulas like:

```text
STDDEV(BTC, 20) > PRICE(BTC) * 0.15
```

### Price expression

```ebnf
price_expr := 'PRICE' '(' symbol ')'
```

Example:

```text
PRICE(MSFT)
PRICE(BTC)
```

### Historical series and TA indicators

```ebnf
series_operation :=
    symbol '.' series_field
  | agg_func '(' symbol ',' NUMBER ')'

series_field := open | high | low | close | volume
agg_func := MAVG | EMA | RSI | STDDEV
```

Examples:

```text
AAPL.close
BTC.volume
RSI(MSFT, 14)
MAVG(AAPL, 50)
STDDEV(BTC, 20)
```

**Design choice (window as NUMBER):** the window argument is `NUMBER` to match the lexer behavior (since `14` is lexed as `NUMBER` anyway). This avoids lexer/parser mismatches.

## Conditions and boolean logic

### Syntax and precedence

```ebnf
condition := disjunction
disjunction := conjunction ('||' conjunction)*
conjunction := (comparison | '(' condition ')') ('&&' conjunction)?
```

Precedence is:

1. comparisons / arithmetic (highest)
2. `&&`
3. `||` (lowest)

So this behaves as expected:

```text
A || B && C     // parsed as: A || (B && C)
```

Parentheses can force grouping:

```text
(A || B) && C
```

## Design choices summary

* **Reserved currencies + `symbol` rule**: solves “BTC is not IDENTIFIER” while keeping currencies protected as reserved words.
* **Optional `IF`**: allows unconditional actions; keeps conditional logic explicit when present.
* **`QTY`**: separates “money-based order” from “quantity-based order”, avoiding ambiguity at runtime.
* **Arithmetic expressions**: operators existed in the lexer already; now the parser supports real numeric formulas.
* **Portfolio allocations match examples**: `value OF symbol` is now a first-class construct.

---

## Current limitations (v0.2)

* No variables, no let/assignments, no named strategies/modules.
* No unary minus (`-x`) currently (you’d add it in `primary`).
* `QTY NUMBER` accepts decimals; if you want integer-only quantities, enforce it lexically or semantically.
* No string literals, no dates/timeframes, no explicit timeframe selection for series.
